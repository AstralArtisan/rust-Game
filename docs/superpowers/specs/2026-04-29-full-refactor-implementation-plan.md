# 勇闯方块城 — 全面重构设计与实施计划

## Context

原项目 `E:/rust_game_merge` 经过多轮迭代（Phase 1-10），代码架构混乱、系统耦合严重。核心问题：
- 奖励房/事件房/商店房逻辑 bug 严重，无法正常使用
- 强化系统只有 2 层，可扩展性差
- 终结技系统解锁条件异常，手感差
- 联机代码是屎山（coop/runtime.rs 单文件过大）
- 状态机扁平设计（13 个顶层状态）导致系统调度混乱
- 大文件问题：enemy/systems.rs 2221行、player/combat.rs 1016行

本次重构采用**方案 B：干净骨架 + 代码移植**，在新 git worktree 中搭建全新架构，从旧代码移植可复用逻辑。

## 1. 工作模式

### 1.1 分支与 Worktree 策略

- 在 `E:/rust_game_merge` 仓库中创建新分支 `full-refactor`
- 使用 `git worktree add` 创建独立工作目录，不动原有分支
- 原 `claude-playground` 分支保持不变，作为旧代码参考

### 1.2 实施策略

**渐进式重构**：能复用就复用，不能复用就重写，保持游戏始终可运行。

预估复用率：
- 玩家移动/朝向：~70%

- 掉落物系统：~60%

需要重写的部分：
- 状态机（扁平 → 三层层级）
- 奖励房/事件房/商店房（逻辑重做）

### 3.2 三层状态机

```rust
#[derive(States, Debug, Clone, Copy, PartialEq, Eq, Hash, Default)]
enum AppState {
    #[default]
    Loading,
    MainMenu,
    InGame,
    CoopLobby,
    CoopGame,
    PvpLobby,
    PvpGame,
    PvpResult,
}

#[derive(SubStates, Debug, Clone, Copy, PartialEq, Eq, Hash, Default)]
#[source(AppState = AppState::InGame)]  // 也适用于 CoopGame
enum GamePhase {
    #[default]
    Playing,
    Paused,
    LevelUp,
    AugmentSelect,
    RewardSelect,
    Shop,
    EventRoom,
    BossChest,
    GameOver,
    Victory,
}

#[derive(SubStates, Debug, Clone, Copy, PartialEq, Eq, Hash, Default)]
#[source(GamePhase = GamePhase::Playing)]
enum RoomPhase {
    #[default]
    Entering,
    Active,
    Cleared,
    Exiting,
}
```

关键改进：UI 覆盖层（暂停、升级、商店等）作为 GamePhase 子状态，不再是顶层 AppState。系统调度更精确。

### 3.3 伤害管线

独立 System 链，通过 Event 传递：

```
碰撞检测 → DamageIntent
  → immunity_check（无敌帧、幽灵态）→ 过滤
  → shield_check（冲刺护盾、词缀护盾）→ 吸收或通过
  → damage_modifier（破甲、方向防御、Boss阶段倍率）→ 修正数值
  → apply_damage → 扣血 → DamageApplied
  → post_damage（击退、吸血、连锁闪电、死亡判定、伤害数字）
```

强化效果通过在对应阶段注入修正 System 来生效，不在战斗代码中硬编码 if-else。

## 4. 强化系统（Augment）

### 4.1 数据结构

```rust
struct AugmentDef {
    id: AugmentId,
    category: AugmentCategory,    // Melee, Ranged, Mobility, General
    rarity: AugmentRarity,        // Common, Elite, Legendary
    title: String,
    levels: [AugmentLevelDef; 3], // 固定3层：基础→进阶→完美
}

struct AugmentLevelDef {
    description: String,
    params: HashMap<String, f32>,  // 数值参数
}

struct AugmentInventory {
    augments: Vec<OwnedAugment>,
}

struct OwnedAugment {
    id: AugmentId,
    level: u8,  // 0=Lv1, 1=Lv2, 2=Lv3
}
```

### 4.2 效果系统

每个强化效果是独立 System，通过 `run_if` 条件控制激活。新增强化只需：
1. 在 `augments.ron` 添加定义
2. 在 `effects.rs` 添加对应 System
3. 在插件中注册

不需要修改战斗核心代码。

### 4.3 获取途径

- 战斗房通关：必定 3 选 1（普通池）
- 精英房通关：3 选 1（精英/传说池）
- 奖励房"强化锻造"：升级已有强化
- 商店：购买强化或升级服务（120 金币）
- 事件房：特定事件可获得
- Boss 宝箱：精英/传说 3 选 1
- 再次获得同一强化：自动升级

## 9. 多人模式

### 9.1 合作模式（Lightyear，主机权威）

核心原则：
- 单机和合作共用同一套游戏逻辑，通过 `SessionMode` 资源区分
- 不在游戏逻辑中散布 if coop 分支，差异通过网络同步层处理
- 合作特殊逻辑封装在 coop/ 模块内

关键设计：
- 掉落物：两人独立拾取同一掉落物，各自获得
- 门选择：各自投票，相同直接进入，不同随机选一个
- 事件房：合作模式保留事件房
- 强化选择：各自独立
- 死亡：变幽灵态，通关当前房间后自动复活 50% HP
- 数值缩放：怪物 HP ×1.6，伤害 ×1.2，数量 +30%

模块拆分（不再是单文件）：
- `protocol.rs`：复制组件、消息、通道定义
- `lobby.rs`：大厅连接、匹配、准备
- `sync.rs`：实体/输入/状态同步
- `authority.rs`：主机权威逻辑
- `visuals.rs`：远程玩家视觉
- `scaling.rs`：合作数值缩放
- `ui.rs`：合作专用 UI

### 9.2 PVP 模式（自写 UDP，轻量原型）

原型功能：两人固定竞技场，近战+远程+冲刺对战，先死者输。
预留扩展空间（道具、地图选择、多回合）。

模块：
- `protocol.rs`：UDP 消息定义
- `net.rs`：网络收发
- `arena.rs`：竞技场逻辑
- `visuals.rs`：对手视觉
- `ui.rs`：PVP 专用 UI
