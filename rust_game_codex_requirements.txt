Rust 2D 小游戏项目需求说明（交给 Codex 的开发任务书）
==================================================

项目名称（暂定）
----------------
《勇闯方块城 / Block City Adventure》

项目目标
--------
使用 Rust 开发一个“2D 俯视角动作 + 轻度 Roguelike + 房间探索 + 轻解谜”的小游戏。
要求游戏画面相对精美、系统设计完整、代码结构清晰，不能只是一个简单 demo。
该项目用于课程大作业，因此除了“能玩”之外，还必须体现：
1. 合理的软件工程结构；
2. 明确的模块划分；
3. 足够的代码复杂度；
4. 可扩展性；
5. 可演示性；
6. 一定的美术表现与交互效果。

一、总体开发要求
----------------
1. 使用 Rust 语言开发。
2. 推荐框架：
   - bevy
   - bevy_rapier2d（碰撞 / 物理）
   - bevy_kira_audio（音频）
   - serde + ron/toml/json（配置驱动）
   - rand（随机生成）
3. 游戏为 2D 俯视角，不做横版跳跃。
4. 必须采用“模块化 + 可维护”的目录结构。
5. 必须使用清晰的状态管理和资源管理。
6. 应尽量体现 ECS 思路（如果使用 Bevy，则自然采用 ECS）。
7. 最终结果必须能运行，至少包含：
   - 主菜单
   - 开始游戏
   - 角色移动
   - 普通攻击
   - 冲刺技能
   - 至少 3 类敌人
   - 房间切换
   - 清怪开门
   - 1 个 Boss
   - 奖励三选一系统
   - HUD/UI
   - 游戏失败与胜利界面
8. 必须包含一定的视觉效果：
   - 角色移动/攻击反馈
   - 怪物受击反馈
   - 简单粒子特效
   - 相机跟随
   - 屏幕震动（至少 Boss 或技能触发时）
   - 房间切换过渡效果
9. 必须保证代码不是“全堆在 main.rs 里”，要拆分成多个模块文件。
10. 尽量保证后续方便继续添加内容。

二、游戏设定与核心玩法
----------------------
游戏类型：
2D 俯视角动作探索 Roguelike。

背景设定：
玩家闯入一座由方块机关、封锁街区与多层试炼区域组成的“方块城”。每一层都由多个房间构成，玩家需要在房间中战斗、探索、解谜、获得强化，并最终击败该层 Boss，前往下一层。

核心循环：
进入楼层 -> 探索房间 -> 清理敌人 / 完成简单机关 -> 获得奖励 -> 强化角色 -> 挑战 Boss -> 进入下一层

核心体验目标：
1. 操作流畅；
2. 战斗有反馈；
3. 有成长感；
4. 房间探索有节奏；
5. UI 和特效让游戏看起来不是简单作业。

三、最小可交付版本（MVP）
-------------------------
必须优先完成以下内容：

A. 基础框架
1. 游戏启动与资源加载；
2. 主菜单；
3. 游戏状态切换；
4. 关卡初始化；
5. 游戏结束界面；
6. 基础音频系统；
7. HUD 显示。

B. 玩家系统
1. WASD / 方向键移动；
2. 鼠标或方向攻击（可任选一种实现方式，但要完整）；
3. 冲刺技能；
4. HP / 能量 / 冷却；
5. 受伤无敌帧；
6. 死亡逻辑；
7. 动画状态切换（至少 Idle / Move / Attack / Dash / Hurt / Dead）。

C. 战斗系统
1. 玩家攻击命中敌人；
2. 敌人攻击命中玩家；
3. 伤害计算；
4. 击退效果；
5. 受击闪烁；
6. 攻击冷却；
7. 子弹或近战范围判定。

D. 敌人系统
至少 3 类敌人：
1. 近战追击型；
2. 远程射击型；
3. 冲锋型；
并至少 1 个 Boss，Boss 要有多阶段逻辑。

E. 房间系统
1. 房间制地图，不做大地图；
2. 初始房、普通战斗房、奖励房、Boss 房；
3. 玩家进入战斗房后关门；
4. 敌人全部消灭后开门；
5. 房间切换时镜头平滑移动或淡入淡出。

F. 奖励系统
每清理一个关键房间后，弹出三选一奖励：
例如：
- 攻击速度提升
- 冲刺冷却缩短
- 最大生命提升
- 暴击率提升
- 击杀回复少量生命
奖励至少实现 6 种。

G. UI 系统
1. 主菜单 UI；
2. 暂停 UI；
3. HUD（血条、技能冷却、当前层数、房间信息）；
4. 奖励选择界面；
5. Boss 血条；
6. Game Over / Victory 界面。

六、必须实现的核心系统设计
--------------------------
1. 状态系统
必须将游戏分为多个状态，而不是一团逻辑混写。
至少包含：
- Loading
- MainMenu
- InGame
- Paused
- RewardSelect
- GameOver
- Victory

2. 实体组件系统（ECS）
所有主要对象都尽量使用组件组织：
- 玩家
- 敌人
- 子弹
- 门
- 陷阱
- 奖励实体
- Boss
- 特效实体

3. 事件系统
推荐通过 Event 分发处理关键行为：
- 伤害事件
- 死亡事件
- 房间清理事件
- 门开启事件
- 奖励选择事件
- Boss 阶段切换事件

4. 配置驱动系统
基础数值不要全硬编码，至少将以下内容配置化：
- 玩家基础属性
- 敌人属性
- Boss 属性
- 奖励属性
- 房间生成参数

5. 战斗反馈系统
必须体现“打击感”：
- 命中时闪白
- 击退
- 音效
- 粒子
- 屏幕震动（适度）

6. 房间推进系统
进入房间后：
- 判断房间类型
- 如果是战斗房，则锁门
- 敌人刷出
- 全灭后发送 RoomClearedEvent
- 开门
- 可进入下一房间

七、推荐的具体玩法规则
----------------------
1. 玩家属性
基础属性：
- HP
- Move Speed
- Attack Power
- Attack Speed
- Crit Chance
- Dash Cooldown
- Energy（可选）

2. 玩家攻击方式
推荐任选一种主方案：
方案 A：近战挥砍 + 冲刺
方案 B：远程能量弹 + 冲刺
方案 C：近战主攻 + 远程技能副手

推荐优先实现方案 A 或 C，因为观感更强。

3. 敌人设计
(1) 近战追击怪
- 持续追玩家
- 接近后近战攻击

(2) 远程射击怪
- 与玩家保持距离
- 周期性发射投射物

(3) 冲锋怪
- 预警短暂停顿
- 直线高速冲锋
- 撞墙后硬直

(4) Boss
建议三阶段：
- 第一阶段：普通追击 + 扇形攻击
- 第二阶段：召唤弹幕 / 范围圈
- 第三阶段：高频冲锋 + 范围爆发

4. 奖励系统
每次奖励三选一，必须随机出现，且尽量不重复。
建议效果：
- 攻速提升 10%~20%
- 生命上限 +20
- 暴击率 +5%
- 冲刺 CD -15%
- 击杀回复 3 HP
- 移速 +10%
- 冲刺附带伤害
- 普通攻击附加一次额外飞弹

5. 房间类型
至少实现：
- Start Room
- Normal Battle Room
- Reward Room
- Boss Room

进阶可加：
- Puzzle Room
- Heal Room
- Elite Room

八、Codex 编码时的具体工程要求
-----------------------------
1. 代码风格要求
- 保持模块清晰；
- 每个文件只负责一个相对明确的功能；
- 避免超长单文件；
- 避免“魔法数字”，尽量提取常量；
- 关键结构体和函数要有注释；
- 尽量保证命名清晰；
- 公共逻辑提取到 utils 或 core；
- 能配置化的尽量配置化。

2. 架构要求
- 使用 plugin/module 组织功能；
- 每个系统可单独注册；
- 系统执行顺序要清楚；
- 状态切换时要有 setup/cleanup；
- UI 和 gameplay 分离；
- 数据定义与运行逻辑分离。

3. 可运行要求
- 生成的代码应能够通过 cargo run 启动；
- 若初期素材不足，可使用占位色块/简单 sprite；
- 先保证玩法完整，再逐步替换美术资源；
- 必须优先保证编译通过与流程可玩。

4. 开发优先级要求
第一优先级：
- 项目骨架
- 状态系统
- 玩家移动
- 简单房间
- 敌人
- 攻击与伤害
- UI 基础

第二优先级：
- Boss
- 奖励系统
- 房间切换
- 特效
- 音频

第三优先级：
- 更复杂地图
- 轻解谜
- 配置驱动优化
- 存档
- 更多动画和内容

十、最重要的技术实现点
--------------------
Codex 在实现时必须特别注意以下几点：

1. 不要把所有逻辑写在同一个系统里；
2. 玩家、敌人、Boss、门、奖励、房间必须模块化；
3. 攻击命中与伤害处理应尽量分离；
4. 房间状态和游戏状态必须明确；
5. 所有 UI 创建后要有 cleanup；
6. RoomCleared、Damage、Death 等应尽量事件化；
7. 代码必须先追求稳定可运行，再追求复杂特效；
8. 素材不足时，可以用彩色方块占位，但架构必须合理；
9. Boss 逻辑不能只是“血量更高的小怪”，必须有阶段变化；
10. 奖励系统必须真的改变玩家战斗表现，而不是只改文本。
